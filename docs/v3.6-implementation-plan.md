# AudioEnhancerMAX v3.6 Implementation Plan

Working title: **Trustworthy Editing**

This plan turns the product roadmap into reviewable implementation slices. The intent is to add editing power without weakening the local-first, explainable, governance-aware position of the project.

## Release Objective

v3.6 should make AudioEnhancerMAX a non-destructive, reviewable audio editor. AI and DSP modules may suggest actions, but the user remains responsible for approving edits, exports, and optional external services.

## Scope Recommendation

### Must Have

1. Non-destructive Edit Decision List
2. AI-assisted Review Dashboard
3. Processing and Governance Report
4. Offline-first Asset Mode
5. Output Profile Manager
6. Transcript-linked editing MVP

### Should Have

7. EDL JSON/CSV export
8. Dependency Doctor improvements

### Not Yet

- Full DAW XML export
- Multi-mic auto-alignment
- Local neural voice restoration
- Short-form video clip generator

These are valuable, but they should follow the editing foundation.

## Resolved Product Decisions

These decisions should be treated as v3.6 design constraints unless product review explicitly reopens them.

1. **Transcript-linked editing belongs in v3.6 as an MVP.** The Whisper word timestamp to EDL relationship must be designed immediately so the data model does not need to be rebuilt in v3.6.1.
2. **Governance reports are backend JSON plus frontend-rendered HTML.** The FastAPI backend owns structured report generation; the frontend renders a readable, printable, and downloadable report view using the existing UI layer.
3. **Offline-first mode disables external badges and scripts.** In offline/no-external mode, the app must not attempt to load SourceForge badge scripts, external fonts, or optional external engines.
4. **EDL v3.6 supports both `cut` and `mute`.** `cut` removes time; `mute` preserves timing and replaces the region with silence, which is often better for breaths, bumps, and short disfluencies.
5. **Output profiles are fixed presets with manual override.** Users can change profile parameters such as target LUFS, but overrides must be recorded in the final report.

## Community Feedback Loop

The public roadmap page should operate as a community signal board, not only as a static planning document.

- Keep v3.6 scope visible as the current implementation foundation.
- Publish a wider sneak peek of future candidates, including DAW export, multi-mic alignment, local neural restoration, short-form clip generation, workflow automation, accessibility, benchmarking, and governance tools.
- Let users signal interest through prefilled GitHub issues and SourceForge comments/reviews.
- Ask for concrete workflow evidence: input files, expected output, hardware, privacy constraints, and willingness to test.
- Build a public ranklist from repeated signals, detailed use cases, sample material, community prototypes, feasibility, and governance fit.
- Treat ranking as prioritization evidence, not as an automatic delivery promise.

## Feature 1: Non-Destructive Edit Decision List

### Goal

Represent edits as structured decisions rather than immediately rewriting media. Original files remain unchanged.

### Proposed Data Model

```json
{
  "schema_version": "1.0",
  "file_id": "abc123",
  "source_path": "app/uploads/abc123.wav",
  "edits": [
    {
      "id": "edit_001",
      "type": "cut",
      "start": 12.42,
      "end": 13.11,
      "reason": "filler_word",
      "source": "review_dashboard",
      "confidence": 0.82,
      "status": "accepted"
    },
    {
      "id": "edit_002",
      "type": "mute",
      "start": 18.20,
      "end": 18.55,
      "reason": "breath",
      "source": "review_dashboard",
      "confidence": 0.74,
      "status": "accepted",
      "fade_ms": 20
    }
  ]
}
```

### v3.6 Edit Types

- `cut`: removes the selected time range from the rendered output.
- `mute`: preserves the timeline duration and replaces the selected region with silence plus short fades.

`attenuate` should remain out of the v3.6 MVP unless implementation proves trivial after `mute`.

### Backend Work

- Add `app/services/edit_decisions.py`.
- Add EDL read/write helpers in `app/outputs/{file_id}_edl.json`.
- Add endpoints:
  - `GET /api/edl/{file_id}`
  - `POST /api/edl/{file_id}/edits`
  - `PATCH /api/edl/{file_id}/edits/{edit_id}`
  - `POST /api/edl/{file_id}/render`

### Frontend Work

- Add timeline state store.
- Show accepted, rejected, and pending edits separately.
- Add undo/redo for EDL operations.

### Acceptance Criteria

- Original upload is not modified.
- Accepted edits can be rendered into a new output file.
- Reopening the same file restores the EDL.
- EDL JSON validates against schema.
- `cut` and `mute` both render correctly and are distinguishable in the report.

### Review Questions

- Should EDL live in `app/outputs/` or a new `app/projects/` directory?
- Should undo/redo be persisted or session-only for v3.6?

## Feature 2: AI-Assisted Review Dashboard

### Goal

Detect likely cleanup opportunities and ask the user to approve them. Avoid blind destructive cleanup.

### Candidate Suggestions

- Filler words
- Long silences
- Breaths
- Mouth sounds
- Stutters
- Hesitations

### Backend Work

- Normalize detector outputs into one suggestion schema.
- Add endpoint:
  - `POST /api/review/suggestions/{file_id}`
- Return transcript references when available.

### Frontend Work

- Add Review panel with filters by suggestion type.
- Add before/after preview for each suggestion.
- Add approve, reject, approve all visible, and reset.
- Show confidence and detection method.

### Acceptance Criteria

- Suggestions do not alter audio until approved.
- Each suggestion has start/end, type, confidence, method, and preview action.
- Bulk approval can be undone.

### Governance Note

The UI should say "suggested edits", not "errors" or "AI corrections".

## Feature 3: Processing and Governance Report

### Goal

Generate a durable record of what happened during processing.

### Report Fields

- App version
- File ID and source metadata
- Selected filters and parameters
- Output profile
- EDL edit count by status and type
- Model/engine used
- Local, LAN, and external service flags
- Timing and benchmark summary
- Output metrics: duration, sample rate, format, peak, loudness where available
- Warnings and limitations

### Backend Work

- Add `app/services/reporting.py`.
- Add endpoint:
  - `GET /api/reports/{file_id}`
  - `POST /api/reports/{file_id}/generate`
- Backend returns canonical JSON only; it should not own HTML report layout.

### Frontend Work

- Add "Download report" button on output panel.
- Add short report summary in About/History if project history exists.
- Render the JSON report as a readable HTML view using existing CSS/JS.
- Provide browser print support and a download option for the raw JSON.

### Acceptance Criteria

- JSON report generated for every render.
- Frontend-rendered HTML report is printable and readable without server-side HTML generation.
- Report includes external-service flags.
- Manual output-profile overrides appear in the report.

## Feature 4: Offline-First Asset Mode

### Goal

Make local/offline behavior verifiable.

### Work Items

- Audit external resources in app pages.
- Bundle required fonts or switch to system fonts in app UI.
- Add setting: `Disable optional external services`.
- When enabled:
  - Hide or disable Edge Neural TTS.
  - Disable SourceForge badge in app context.
  - Prevent external badge/script/font loading attempts.
  - Report `external_services_allowed: false`.

### Acceptance Criteria

- App UI loads without Google Fonts.
- Offline mode prevents optional external TTS.
- Offline mode does not request SourceForge badge scripts or other external UI assets.
- Processing report states no external services were used when true.

## Feature 5: Output Profile Manager

### Goal

Make export behavior explicit and repeatable.

### Initial Profiles

- Podcast: WAV/MP3, target -16 LUFS
- YouTube: WAV, target -14 LUFS
- Audiobook: WAV, target -18 LUFS
- Archive: WAV/FLAC, preserve sample rate
- Preserve Original: keep source sample rate when possible

### Backend Work

- Add output profile schema.
- Map profile to existing `ProcessingOptions`.
- Add endpoint:
  - `GET /api/output-profiles`
- Accept manual overrides and preserve them separately from the base profile.

### Frontend Work

- Add profile selector near output format controls.
- Show what the profile changes before processing.
- Allow manual override of selected profile values.

### Acceptance Criteria

- Output format and loudness are visible before render.
- Profile is recorded in the governance report.
- User override is possible, marked as an override, and recorded in the report.

## Feature 6: Transcript-Linked Editing MVP

### Goal

Create EDL edits from transcript selections.

### Dependencies

- EDL engine
- Streaming transcript with word timestamps
- Review dashboard UI patterns

### MVP

- Select transcript word range.
- Create cut or mute edit.
- Preview before render.
- Store as EDL.
- Persist `word_start`, `word_end`, transcript text, and original Whisper timestamps in the EDL edit metadata.

### Acceptance Criteria

- Timestamp mapping is visible.
- User can reject generated edit.
- Source transcript is not treated as ground truth without review.
- Transcript-linked edits can generate both `cut` and `mute` decisions.
- EDL schema supports future transcript improvements without changing core edit fields.

## Remaining Decisions for Product Review

1. Should EDL project files live in `app/outputs/` for simplicity or a new `app/projects/` directory for long-term project management?
2. Should undo/redo history be persisted in v3.6 or kept session-only?
3. Which fixed output profiles are required for the first release: Podcast, YouTube, Audiobook, Archive, Preserve Original, or fewer?
4. Should report JSON be attached automatically to every output download, or generated on demand?
5. Should offline-first mode be a global app setting, a per-session setting, or both?

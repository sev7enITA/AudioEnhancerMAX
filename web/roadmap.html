Candidate scope

v3.6 builds the editing foundation

These items are prioritized because they improve control, auditability, and creator workflow without promising blind automation.

Foundation P0/P1

Non-Destructive Edit Decision List

Represent cuts, mutes, fades, transcript-linked selections, and accepted suggestions as structured timeline decisions while preserving the original uploaded media.

BenefitUndo, redo, re-render, and audit each processing run.
ReviewEDL JSON schema, cut/mute rendering, playback accuracy, export parity.
Transcript P1

Transcript-Linked Editing

Create cut or mute timeline edits by selecting transcript words or ranges. The MVP belongs in v3.6 so Whisper word timestamps and EDL metadata are designed together.

BenefitFaster trimming while preserving human review and reversibility.
ReviewWord timestamp precision, transcript metadata in EDL, preview playback.
Editing UX P1

AI-Assisted Review Dashboard

Surface filler words, long silences, breaths, mouth sounds, and stutters as suggestions on transcript and waveform instead of applying blind destructive cuts.

BenefitFaster cleanup while reducing choppy or over-processed results.
ReviewAccept, reject, preview, bulk approve, confidence labels.
Governance P1

Processing and Governance Report

Generate canonical JSON for each export; render the readable HTML report in the frontend with selected filters, parameters, model use, timing, warnings, and data-flow notes.

BenefitAn audit trail for professional, research, and governance-sensitive workflows.
ReviewBackend JSON, frontend-rendered HTML, print support, raw JSON download.
Privacy P1

Offline-First Asset Mode

Bundle runtime UI assets locally and expose a mode that disables optional external services, external fonts, and badge scripts for sessions where no-external operation matters.

BenefitClearer privacy posture for field work and restricted networks.
ReviewNetwork call audit, service toggles, disabled SourceForge badge/script loading.
Export P1

Output Profile Manager

Define fixed podcast, YouTube, audiobook, archival, and preserve-original profiles with manual overrides recorded in the final report.

BenefitRemoves guesswork and prevents unexpected format changes.
ReviewFixed schemas, override markers, report integration.

How user preference becomes a ranklist

The ranklist should guide prioritization, not replace technical judgment. A popular feature still needs feasibility, governance review, and measurable acceptance criteria.

01

Signal interest

Click a roadmap card to open a prefilled GitHub issue. Users can also comment on SourceForge using the feature name.

02

Describe the workflow

The most useful signals include input type, expected output, hardware, privacy constraints, and why the current app does not solve it yet.

03

Collect evidence

Repeated requests, concrete examples, test files, and community prototypes move an idea higher than generic interest alone.

04

Publish the shortlist

Release planning can publish the top candidates with feasibility notes, risks, and what must be validated before implementation.

Implementation discipline

How v3.6 should be reviewed

Each feature should pass product review, technical review, and governance review before it becomes part of public release notes.

Step 1

Define schemas and review gates

Lock EDL JSON, processing report JSON, output profile schema, transcript timestamp mapping, and external-service flags before building UI.

Step 2

Build minimum reviewable workflow

Upload, transcribe, detect suggestions, preview cut/mute edits, accept/reject, export final audio, and generate the report.

Step 3

Measure before promising

Track duration drift, loudness, peak, sample rate, accepted suggestion count, model used, profile overrides, and whether any external service was enabled.

Step 4

Publish only defensible claims

Release notes should say what is implemented, what is experimental, and what still requires human review.
